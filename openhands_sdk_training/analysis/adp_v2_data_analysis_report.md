# ADP-v2 Data Analysis: Which sources to train a 4B SWE agent on

Compiled 2026-07-13; §5–6 updated 2026-07-15 to reflect the as-launched four-arm campaign and the
model-souping plan. Companion data: [`v2_sft_action_analysis.json`](v2_sft_action_analysis.json)
(behavioral stats), generated by
[`scripts/analyze_v2_sft_actions.py`](../scripts/analyze_v2_sft_actions.py).

## 1. Motivation

Graham's Qwen3.5-4B SFT on the v1 `full_condenser_24k_all_records` mixture *regressed* on
SWE-bench Verified vs. the untouched base model: **14/500 resolved (fine-tuned, ckpt-2000) vs
25/500 (base)** at temp 0, with **84% empty patches**. Two confounds: (a) the checkpoint was only
~⅓ of one epoch (2-day TIMEOUT at step ~1697–2000 of 5322, so evaluated mid-cosine-decay at ~70%
of peak LR — never annealed), and (b) the data mixture was dominated by non-editing / off-domain
sources.

Last week's runs already showed data recipe is the lever: a curated v1 mixture
(**paper-nonweb, full epoch → 52/500 = 10.4%**) beat both base (25) and Graham's diluted run (14).
The `neulab/adp-v2` release (6.87M records across 50 configs, 24k-token condenser SFT split) is the
scaled-up, more-on-domain continuation of that pipeline. This report characterizes which v2 sources
actually teach SWE editing, and their provenance/quality, to design the next arms.

## 2. Behavioral action analysis

**Method.** Streamed head+mid windows (~500 records/config) of each config's `sft_openhands_24k`
JSONL and classified, per record: final-turn action; whether any file-edit occurs (`edit%`);
whether an edit occurs in the last 3 assistant turns (`ed<3%`, the proxy for "ends by producing a
patch"); test/verification evidence (`test%`); and the completion style (`finish%`).

**Important classifier note.** Edit detection must recognize **SWE-agent ACI syntax**
(`edit <s>:<e> … end_of_edit`, `create <path>`) passed as the `command` arg to the `terminal` tool —
ADP does *not* rewrite these into OpenHands `file_editor` calls. A first pass missed this and
reported `nebius_SWE-agent` at 1.4% edit; corrected, it is ~76% edit / 66% ed<3. `finish%` is a
*scaffold* signal, not quality: SWE-agent / mini-swe-agent sources submit implicitly (the harness
extracts the diff), so `finish%=0` there is expected, not "gives up." Trust `edit%` / `ed<3%` /
`test%`.

**Old-mix contrast (why the negative result happened).** The v1 all-records mix was heavily diluted
with 0%-edit conversational sources: `orca_agentinstruct`, `synatra`, `codeactinstruct`,

`nebius_SWE-agent` 0.2, and *upsample* `swe-gym` to 3.0 — arrived at by hand, matching this analysis.

**Tiers** (for a SWE-bench-targeted, OpenHands-evaluated 4B SFT):
- **T1 — verified + OpenHands + strong teacher:** coderforge, scale_swe_distilled, nebius_SWE-rebench.
- **T2 — verified, scaffold/task caveats:** logicstar & mini-coder (mini-swe-agent); swe-smith
  (SWE-agent); swe-gym & hybrid-gym (OpenHands but tiny / off-task).
- **T3 — downweight/drop:** SWE-Zero & nebius_SWE-agent (unverified), kwai (teacher unknown),
  openthoughts / SALT / codescout (off-target).

**Open provenance items:** `swe-gym` has an ADP source-URL inconsistency (may be the unfiltered
verifier set — verify via `extract_raw.py`); `logicstar` license unstated; hybrid-gym & swe-play are
CMU/All-Hands preprints (dfried is a coauthor). Teacher models skew Claude-family + Qwen3-Coder-480B
across the strong configs. Scaling context: the ADP paper only studies 7B/14B/32B (OpenHands+ADP →
20.4/30.6/36.8% on SWE-bench Verified); 4B sits below that range, consistent with our 10.4%.

## 4. Throughput & sizing (measured, 4×L40S)

From last week's trainer states (`trainer_state.json` on tir1) + sacct:

| run | GPUs | steps | runtime | s/step | samples/s | tok/sample | tok/s agg |
|---|--:|--:|--:|--:|--:|--:|--:|
| swe-smith Base | 8×L40S | 540 | 13.1h | 87.2 | 0.367 | 24,835 | ~9,100 |
| swe-smith instruct | 4×L40S | 540 | 13.3h | 88.9 | 0.360 | 24,835 | ~8,900 |
| paper-nonweb Base | 4×L40S | 1154 | 25.7h | 80.3 | 0.398 | 5,943 | ~2,370 |

Two findings that govern sizing:
1. **Step/sample-bound, not token-bound.** samples/s ≈ 0.36–0.40 regardless of sequence length
   (ZeRO-3 param all-gather over PCIe dominates at bs1). **Training time scales with RECORD COUNT,
   ~independent of sequence length.** Use records, not tokens, as the sizing unit.
2. **8 GPU ≈ 4 GPU** (no NVLink on L40S): doubling GPUs gave ~no speedup. 4-GPU is the efficient shape.

**Planning rule: ~0.38 samples/s ≈ ~32–33K records/day on 4×L40S.** One epoch:

| records | epoch time | 2-day jobs |
|---:|---:|---:|
| 170K (old-run size) | ~5 days | ~3 |
| 180K | ~5.6 days | ~3 |
| 200K | ~6.25 days | ~4 |
| 250K | ~7.8 days | ~5 |
| 2.63M (all SWE) | ~80 days | infeasible |
| 6.87M (full 24k split) | ~210 days | infeasible |

**Any v2 run must subsample.** For a Sunday deadline launching Monday night (~6-day window minus
prep), **~180K records = one full epoch** is the safe target; 250K would slip to ~Tuesday. A partial
epoch must be avoided — it reintroduces the exact LR-schedule confound (§1) the campaign exists to
remove.

## 5. Launched experiments (four single-source arms, 2026-07-13/14)

The launched design supersedes the earlier two-arm 180K plan: **four single-source arms at 55K
records each** instead of two at 180K. Same total GPU budget, but every Tier-1 source (plus the
unverified contrast) gets its own model, and because all four share the same init their task
vectors are directly mergeable — the data-mixture question moves from "train every mix" to a cheap
weight-space coefficient search (§6).

**Recipe (all arms identical):** instruct-init `Qwen/Qwen3.5-4B`, full SFT, seq 32768, 4×L40S,
ZeRO-3+FA2+Liger, 1 epoch over 55K records (`max_samples`, ≈1719 steps, ~42h/arm). Training data
kept in **native OpenHands-SDK OpenAI format** (matches the eval agent) rather than down-converting
to sharegpt. Run dirs `~/exp/adp-smoke/runs/v2_<arm>_inst_4b/`, outputs + datasets on tir1.

| arm | source | verified? | teacher | role |
|---|---|:--:|---|---|
| coderforge | coderforge_preview 55K | ✅ | Qwen3-Coder-480B | T1; anchor of verification contrast |
| swezero | nvidia_SWE-Zero 55K | ❌ | Qwen3-Coder-480B | **verification contrast** vs coderforge (same teacher/scaffold/domain, differs ~only in resolution-filtering) |
| scale | scale_swe_distilled 55K | ✅ | DeepSeek v3.2 | T1; teacher-diversity |
| rebench | nebius_SWE-rebench 55K | ✅ | Qwen3-Coder-480B | T1; +regression-tests filtering |

**Comparisons:** base = 25/500, paper-nonweb (v1 curated, Base-init) = 52/500; a raw-instruct
baseline eval is in flight so the instruct-init arms have a matched untrained anchor.

**Status 2026-07-15:** coderforge **done** (train loss 0.29; swe-gym val plateau ~0.544 vs base
anchor 0.5927) with its full-500 SWE-bench eval running as chained legs; swezero training on
`general` (resumed from ck-200); scale/rebench pending on `preempt`. Arms auto-requeue on
preemption with resume-from-latest-checkpoint (`save_steps: 25` on preempt arms), plus a login2
watchdog that resubmits on crash/timeout.

**Note on val loss:** all arms drop below the base swe-gym val anchor early (~0.52–0.53) then
drift up to a plateau — off-policy token CE has a teacher-mismatch floor, so val loss is a sanity
signal only. Data quality is judged by SWE-bench resolve / empty-patch rate; this also rules out
val loss as the search signal for souping (§6).

## 6. Souping plan: coefficient search over the 4 task vectors

All arms share init θ₀, so each defines a task vector τᵢ = θᵢ − θ₀ and any soup is
θ(w) = θ₀ + Σ wᵢτᵢ — a linear weighted average of state dicts (a small safetensors script; no
mergekit needed for linear merges). Record counts are matched at 55K, so vector norms are
comparable without per-arm normalization. Merging is minutes; **the binding cost is that every
soup point needs a SWE-bench subset eval**, which drives the design below.

**Eval substrate:** a pinned **150-instance subset** of SWE-bench Verified, drawn from the same
500 rows as coderforge's full run (so coderforge's anchor comes free, and every point is scored on
identical instances → paired per-instance comparisons recover statistical power). Prereq: build
the subset runner — `full/run_full_infer.sbatch` is full-split-only and the smoke script caps at 20.

**Design — uniform + leave-one-out (~10–11 subset evals total):**
1. **Anchors (3 evals):** individual subset scores for scale / rebench / swezero (coderforge free).
2. **Uniform soup** (¼,¼,¼,¼) — the standard strong baseline (1 eval).
3. **Four leave-one-out soups** (⅓ each of the remaining three). Marginal contribution of source
   i = uniform − LOOᵢ: a Shapley-style readout of per-source value that doubles as the weight-space
   replication of the verification contrast, and automatically answers whether swezero belongs in
   the mix at all (4 evals).
4. **Global scale check:** θ₀ + λ·mean(τ) for λ ∈ {0.6, 1.3} — task arithmetic often peaks off
   λ=1 (1–2 evals).
5. **Final soup:** w ∝ max(marginal contribution, 0); evaluate, then confirm the winner on
   **full 500** for a headline number comparable to base/paper-nonweb/coderforge-full (1 eval + full run).

**Power caveat:** at ~12% resolve, 150 instances give ~±2.7% SE per point, so individual LOO
deltas under ~5 points are noise-level in isolation — treat LOO as a *ranking* signal (helped by
pairing), and let the full-500 confirmation carry the headline claim.

**Confirmation train:** one combined run on the soup-derived mixture at **~180–200K records**
(mix proportions ∝ final w), i.e. ~4× the per-arm budget: ~6 days on 4×L40S as ~4 chained 2-day
legs with checkpoint/resume. The four 55K arms already provide the matched-budget science; the
scaled run is the best-final-model play. Caveat to keep in mind when reading the result: soup
weights are a proxy for data-mix weights (averaging functions ≠ joint training), so the scaled run
is also the test of that proxy.

**Supporting literature.** Direct precedent for the coefficients-as-mixture-weights step:
**DEM** (Ram et al., arXiv:2406.15570, EMNLP 2024) trains per-distribution models from a shared
init and combines distribution vectors with validation-tuned coefficients as a substitute for data
mixing; **DeMix** (Li et al., arXiv:2602.00747) makes the readout explicit — merge coefficients
over per-source component models set the mixture ratios for the subsequent real training run (see
also MergeMix, arXiv:2601.17858, and arXiv:2602.04937 for the same idea in multimodal mixing);
**Branch-Train-Merge** (Li et al., arXiv:2208.03306) established the parallel
per-domain-experts-then-merge paradigm for LMs. The merge operation itself: **Model Soups**
(Wortsman et al., arXiv:2203.05482) for uniform/greedy averaging of shared-init fine-tunes;
**task arithmetic** (Ilharco et al., arXiv:2212.04089) for additive task vectors and the global-λ
scaling (often peaking at λ≠1 — our step 4); **TIES** (Yadav et al., arXiv:2306.01708) and
**DARE** (Yu et al., arXiv:2311.03099) as fallbacks if sign-interference hurts plain linear soups
(unlikely here — same-task, same-domain fine-tunes). Same-init averaging is sound because
fine-tunes from a shared init stay linearly mode-connected (Frankle et al., arXiv:1912.05671;
Neyshabur et al., arXiv:2008.11687) — our matched-init, matched-55K arms are deliberately inside
that regime. The non-souping comparison class — **DoReMi** (Xie et al., arXiv:2305.10429) and
**RegMix** (Liu et al., arXiv:2407.01492) — derives mixture weights from cheap proxy *trainings*;
we substitute merges for proxy runs, which is precisely the trade DEM/DeMix validate.

## 7. Limitations

- Behavioral stats are ~500-record samples per config with a regex classifier — treat percentages as
  ±a few points.
- Provenance verified largely by upstream row-count matches, not an explicit adp-v2 provenance table
  (the card has none); swe-gym filtering and logicstar license remain open.
- tok/rec for most configs is the ~16k planning estimate; measured directly only for a subset
  (swe-smith 19.5k, SWE-Zero 18.6k, coderforge 14.6k, logicstar 16.1k, nebius-agent 12.9k).
- Throughput is from L40S; A100/H100 would differ (and A100 has NVLink → better multi-GPU scaling).
