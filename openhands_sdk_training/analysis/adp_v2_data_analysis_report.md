# ADP-v2 Data Analysis: Which sources to train a 4B SWE agent on

Compiled 2026-07-13. Companion data: [`v2_sft_action_analysis.json`](v2_sft_action_analysis.json)
(behavioral stats), generated by
[`scripts/analyze_v2_sft_actions.py`](../scripts/analyze_v2_sft_actions.py).

## 1. Motivation

Graham's Qwen3.5-4B SFT on the v1 `full_condenser_24k_all_records` mixture *regressed* on
SWE-bench Verified vs. the untouched base model: **14/500 resolved (fine-tuned, ckpt-2000) vs
25/500 (base)** at temp 0, with **84% empty patches**. Two confounds: (a) the checkpoint was only
~⅓ of one epoch (2-day TIMEOUT at step ~1697–2000 of 5322, so evaluated mid-cosine-decay at ~70%
of peak LR — never annealed), and (b) the data mixture was dominated by non-editing / off-domain
sources.

Last week's runs already showed data recipe is the lever: a curated v1 mixture
(**paper-nonweb, full epoch → 52/500 = 10.4%**) beat both base (25) and Graham's diluted run (14).
The `neulab/adp-v2` release (6.87M records across 50 configs, 24k-token condenser SFT split) is the
scaled-up, more-on-domain continuation of that pipeline. This report characterizes which v2 sources
actually teach SWE editing, and their provenance/quality, to design the next arms.

## 2. Behavioral action analysis

**Method.** Streamed head+mid windows (~500 records/config) of each config's `sft_openhands_24k`
JSONL and classified, per record: final-turn action; whether any file-edit occurs (`edit%`);
whether an edit occurs in the last 3 assistant turns (`ed<3%`, the proxy for "ends by producing a
patch"); test/verification evidence (`test%`); and the completion style (`finish%`).

**Important classifier note.** Edit detection must recognize **SWE-agent ACI syntax**
(`edit <s>:<e> … end_of_edit`, `create <path>`) passed as the `command` arg to the `terminal` tool —
ADP does *not* rewrite these into OpenHands `file_editor` calls. A first pass missed this and
reported `nebius_SWE-agent` at 1.4% edit; corrected, it is ~76% edit / 66% ed<3. `finish%` is a
*scaffold* signal, not quality: SWE-agent / mini-swe-agent sources submit implicitly (the harness
extracts the diff), so `finish%=0` there is expected, not "gives up." Trust `edit%` / `ed<3%` /
`test%`.

**Old-mix contrast (why the negative result happened).** The v1 all-records mix was heavily diluted
with 0%-edit conversational sources: `orca_agentinstruct`, `synatra`, `codeactinstruct`,
`code_feedback` all ~0% edit, 3–8 messages. Only swe-smith/swe-gym taught repo editing. This matches
the 84% empty-patch failure. **The ADP paper (v1) independently confirms this**: its own per-dataset
sampling multipliers (Table 9) downweight `orca` to 0.001, `synatra` 0.01, `code_feedback` 0.1,
`nebius_SWE-agent` 0.2, and *upsample* `swe-gym` to 3.0 — arrived at by hand, matching this analysis.

**Tiers** (for a SWE-bench-targeted, OpenHands-evaluated 4B SFT):
- **T1 — verified + OpenHands + strong teacher:** coderforge, scale_swe_distilled, nebius_SWE-rebench.
- **T2 — verified, scaffold/task caveats:** logicstar & mini-coder (mini-swe-agent); swe-smith
  (SWE-agent); swe-gym & hybrid-gym (OpenHands but tiny / off-task).
- **T3 — downweight/drop:** SWE-Zero & nebius_SWE-agent (unverified), kwai (teacher unknown),
  openthoughts / SALT / codescout (off-target).

**Open provenance items:** `swe-gym` has an ADP source-URL inconsistency (may be the unfiltered
verifier set — verify via `extract_raw.py`); `logicstar` license unstated; hybrid-gym & swe-play are
CMU/All-Hands preprints (dfried is a coauthor). Teacher models skew Claude-family + Qwen3-Coder-480B
across the strong configs. Scaling context: the ADP paper only studies 7B/14B/32B (OpenHands+ADP →
20.4/30.6/36.8% on SWE-bench Verified); 4B sits below that range, consistent with our 10.4%.

## 4. Throughput & sizing (measured, 4×L40S)

From last week's trainer states (`trainer_state.json` on tir1) + sacct:

| run | GPUs | steps | runtime | s/step | samples/s | tok/sample | tok/s agg |
|---|--:|--:|--:|--:|--:|--:|--:|
| swe-smith Base | 8×L40S | 540 | 13.1h | 87.2 | 0.367 | 24,835 | ~9,100 |
| swe-smith instruct | 4×L40S | 540 | 13.3h | 88.9 | 0.360 | 24,835 | ~8,900 |
| paper-nonweb Base | 4×L40S | 1154 | 25.7h | 80.3 | 0.398 | 5,943 | ~2,370 |

Two findings that govern sizing:
1. **Step/sample-bound, not token-bound.** samples/s ≈ 0.36–0.40 regardless of sequence length
   (ZeRO-3 param all-gather over PCIe dominates at bs1). **Training time scales with RECORD COUNT,
   ~independent of sequence length.** Use records, not tokens, as the sizing unit.
2. **8 GPU ≈ 4 GPU** (no NVLink on L40S): doubling GPUs gave ~no speedup. 4-GPU is the efficient shape.

**Planning rule: ~0.38 samples/s ≈ ~32–33K records/day on 4×L40S.** One epoch:

| records | epoch time | 2-day jobs |
|---:|---:|---:|
| 170K (old-run size) | ~5 days | ~3 |
| 180K | ~5.6 days | ~3 |
| 200K | ~6.25 days | ~4 |
| 250K | ~7.8 days | ~5 |
| 2.63M (all SWE) | ~80 days | infeasible |
| 6.87M (full 24k split) | ~210 days | infeasible |

**Any v2 run must subsample.** For a Sunday deadline launching Monday night (~6-day window minus
prep), **~180K records = one full epoch** is the safe target; 250K would slip to ~Tuesday. A partial
epoch must be avoided — it reintroduces the exact LR-schedule confound (§1) the campaign exists to
remove.

## 5. Recommended experiments (two 4-GPU arms in parallel)

Same recipe as last week (Base-init Qwen3.5-4B, seq 32768, ZeRO-3+FA2+Liger, lr 1e-5 cosine,
in-training eval), same ~180K / one-epoch size, evaluated on the validated
`~/exp/adp-smoke/swebench/full/` harness. Both compare against existing **base=25/500** and
**paper-nonweb=52/500**.

**Primary — "does trajectory verification matter?" (controlled)**
- Slot A: **coderforge_preview 180K** (verified)
- Slot B: **nvidia_SWE-Zero 180K** (unverified)
- These are a near-perfect natural experiment: same teacher (Qwen3-Coder-480B), same OpenHands
  scaffold, same domain (real GitHub issues), differing ~only in resolution-verification. If A ≫ B,
  future curation must resolution-filter (and SWE-Zero's 956K is a trap despite its size); if A ≈ B,
  unverified scale is cheap. Both single-source → no mixing confound.

**Alternative — maximize the best single result**
- Slot A: curated Tier-1 mix — coderforge 70K + scale 60K + rebench 50K (all verified, all OpenHands).
- Slot B: coderforge solo 180K → tests "diversity vs single best verified source."

**Corrections to earlier plans:** SWE-Zero is the one *unverified* big source, not "highest quality";
`nebius_SWE-agent` is ~83% failed trajectories and needs `resolved`-field filtering before use.

**Logistics:** two 4-GPU trains = the full 8-GPU `general` cap, so park/move eval jobs to `preempt`
first; prep both tokenized datasets tonight (download → `sft_to_llamafactory` → pre-tokenize); expect
~3 chained 48h legs with checkpoint/resume per arm.

## 6. Limitations

- Behavioral stats are ~500-record samples per config with a regex classifier — treat percentages as
  ±a few points.
- Provenance verified largely by upstream row-count matches, not an explicit adp-v2 provenance table
  (the card has none); swe-gym filtering and logicstar license remain open.
- tok/rec for most configs is the ~16k planning estimate; measured directly only for a subset
  (swe-smith 19.5k, SWE-Zero 18.6k, coderforge 14.6k, logicstar 16.1k, nebius-agent 12.9k).
- Throughput is from L40S; A100/H100 would differ (and A100 has NVLink → better multi-GPU scaling).
